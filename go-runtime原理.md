### 1.2 编程语言分类

#### 按执行方式分类

```
源代码
   │
   ├──────────────────────────────────────┐
   ↓                                      ↓
┌─────────────────┐              ┌─────────────────┐
│   编译型语言     │              │   解释型语言     │
│  (Compiled)     │              │  (Interpreted)  │

   ┌─────┴─────┐                          │
   ↓           ↓                          ↓
```

| 类型 | 执行链路 | 代表语言 | 特点 |
|------|----------|----------|------|
| **原生编译型** | 源码 → 机器码 → CPU | C, C++, Rust, **Go** | 性能最高，运行时开销小 |

| **混合型** | 源码 → 字节码 → JIT → CPU | JavaScript, PyPy | 兼顾开发效率和运行性能 |

## 第二章：Go Runtime 是什么

### 2.1 定义

Go Runtime 是 Go 语言的运行时系统，**编译时直接嵌入到可执行文件中**，负责：
- Goroutine 调度
- 内存分配与垃圾回收
- 栈管理
- Channel 通信
- 系统调用封装

### 2.2 与虚拟机语言的本质区别

| 维度 | 虚拟机语言（Java/C#） | Go Runtime |
|------|----------------------|------------|
| 执行模型 | 解释执行字节码 + JIT | **AOT 编译为原生机器码** |
| 部署形态 | 独立运行的虚拟机进程 | 嵌入可执行文件 |
| 跨平台方式 | 字节码跨平台 | 交叉编译生成目标平台二进制 |
| 启动速度 | 较慢（类加载、JIT 预热） | **极快**（无预热） |
| 内存占用 | 较大（VM 本身开销） | 较小 |
| 运行时优化 | JIT 热点编译 | 编译期静态优化 |

### 2.3 类比理解

```
JVM/CLR ≈ VMware（虚拟完整机器环境）
Docker ≈ 虚拟操作系统
Go Runtime ≈ 轻量级运行时库（类似 C 语言的 libc）
```

Go 不存在"虚拟机"概念，Runtime 更接近于一个**高度优化的并发运行时库**。

## 第三章：程序加载与初始化

### 3.1 加载过程

Go 程序启动流程（对标 JVM 类加载）：

```
1. OS 加载可执行文件到内存
2. 执行 runtime 初始化代码
   ├── 初始化调度器（创建 M0、G0）
   ├── 初始化内存分配器
   ├── 初始化垃圾回收器
   └── 设置信号处理
3. 创建 main goroutine
4. 执行 init() 函数链
5. 执行 main.main()
```

### 3.2 init() 函数

- 每个包可有多个 `init()` 函数
- 按依赖顺序执行（被依赖的包先初始化）
- 同包内按文件名字母序、同文件内按声明顺序执行

```go
// 类似 JVM 的 <clinit> 静态初始化块
func init() {
    // 包级别初始化逻辑
}
```

### 3.3 链接与符号解析

Go 使用**静态链接**（默认），符号在编译期解析完成，无需运行时解析（对比 JVM 的符号引用→直接引用）。

## 3. GMP 调度模型（核心）

### 3.1 三大组件

| 组件 | 全称 | 说明 |
|------|------|------|
| **G** | Goroutine | 轻量级协程，初始栈 2KB（线程通常 1MB+） |
| **M** | Machine | 操作系统线程，执行 G 的载体 |
| **P** | Processor | 逻辑处理器，持有本地运行队列 |

### 3.2 调度架构

```
              全局队列 (Global Run Queue)
                       │
        ┌──────────────┼──────────────┐
        ▼              ▼              ▼
   ┌─────────┐    ┌─────────┐    ┌─────────┐
   │   P0    │    │   P1    │    │   P2    │
   │ ┌─────┐ │    │ ┌─────┐ │    │ ┌─────┐ │
   │ │Local│ │    │ │Local│ │    │ │Local│ │
   │ │Queue│ │    │ │Queue│ │    │ │Queue│ │
   │ └─────┘ │    │ └─────┘ │    │ └─────┘ │
   └────┬────┘    └────┬────┘    └────┬────┘
        │              │              │
        ▼              ▼              ▼
   ┌─────────┐    ┌─────────┐    ┌─────────┐
   │   M0    │    │   M1    │    │   M2    │
   │ (Thread)│    │ (Thread)│    │ (Thread)│
   └─────────┘    └─────────┘    └─────────┘
```

### 3.3 调度策略

**获取 G 的优先级**：
1. P 本地队列（无锁访问）
2. 全局队列（需加锁，每 61 次调度检查一次）
3. 网络轮询器（netpoller）
4. **工作窃取**（从其他 P 偷取一半 G）

**抢占机制**：
- Go 1.14+ 支持**基于信号的异步抢占**
- 长时间运行的 G 会被强制让出 CPU（防止饥饿）

### 3.4 G 的状态流转

```
        ┌─────────────────────────────────┐
        ▼                                 │
   ┌─────────┐    go func()    ┌─────────┐│
   │ _Gidle  │ ───────────────▶│_Grunnable││
   └─────────┘                 └────┬────┘│
                                    │     │
                              调度执行     │
                                    ▼     │
                              ┌─────────┐ │
                              │_Grunning│ │
                              └────┬────┘ │
                    ┌──────────────┼──────┴───────┐
                    ▼              ▼              ▼
              ┌─────────┐   ┌──────────┐   ┌─────────┐
              │_Gwaiting│   │_Gsyscall │   │ _Gdead  │
              │ (阻塞)   │   │ (系统调用)│   │ (结束)  │
              └─────────┘   └──────────┘   └─────────┘
```

### 3.5 与 JVM 线程模型对比

| 维度 | JVM 线程 | Go Goroutine |
|------|----------|--------------|
| 对应关系 | 1:1（Java线程:OS线程） | M:N（多G:多M） |
| 创建开销 | 约 1MB 栈内存 | 约 2KB（可动态增长） |
| 切换开销 | 内核态切换（微秒级） | 用户态切换（纳秒级） |
| 调度器 | 操作系统 | Go Runtime |
| 数量上限 | 数千 | 数百万 |

## 4. 内存模型

### 4.1 内存区域划分

```
┌──────────────────────────────────────────────────┐
│                    进程地址空间                    │
├──────────────────────────────────────────────────┤
│  ┌────────────────────────────────────────────┐  │
│  │              Heap（堆）                     │  │
│  │  ┌─────────┐ ┌─────────┐ ┌─────────────┐  │  │
│  │  │ mspan   │ │ mspan   │ │   mspan     │  │  │
│  │  │ (小对象) │ │ (中对象) │ │  (大对象)   │  │  │
│  │  └─────────┘ └─────────┘ └─────────────┘  │  │
│  └────────────────────────────────────────────┘  │
│                                                  │
│  ┌────────────────────────────────────────────┐  │
│  │           Goroutine Stacks（栈）            │  │
│  │  ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐  │  │
│  │  │ G1  │ │ G2  │ │ G3  │ │ G4  │ │ ... │  │  │
│  │  └─────┘ └─────┘ └─────┘ └─────┘ └─────┘  │  │
│  └────────────────────────────────────────────┘  │
│                                                  │
│  ┌────────────────────────────────────────────┐  │
│  │         Runtime 数据结构                    │  │
│  │  mcache | mcentral | mheap | 调度器状态    │  │
│  └────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────┘
```

### 4.2 内存分配器（TCMalloc 变体）

**三级缓存架构**：

| 层级 | 组件 | 说明 |
|------|------|------|
| L1 | **mcache** | 每个 P 私有，无锁分配 |
| L2 | **mcentral** | 按 size class 分类，需加锁 |
| L3 | **mheap** | 全局堆，管理 arena（64MB 块） |

**对象大小分类**：

| 类型 | 大小 | 分配方式 |
|------|------|----------|
| 微对象 | <16B | 合并分配到同一内存块 |
| 小对象 | 16B~32KB | mcache → mcentral |
| 大对象 | >32KB | 直接从 mheap 分配 |

### 4.3 栈内存管理

- **初始大小**：2KB
- **动态增长**：函数序言检查，不足时扩容（连续栈，拷贝迁移）
- **栈收缩**：GC 时检测，缩小为原来的一半

```go
// 栈增长触发条件（编译器插入的检查）
if SP < stackguard {
    runtime.morestack()
}
```

### 4.4 与 JVM 内存区域对比

| JVM | Go Runtime | 说明 |
|-----|------------|------|
| 堆（Heap） | mheap | 对象分配 |
| 虚拟机栈 | Goroutine Stack | 函数调用栈 |
| 方法区/元空间 | 编译时确定（嵌入二进制） | 类型信息、常量 |
| 程序计数器 | G 结构体中的 pc 字段 | 当前执行位置 |
| 本地方法栈 | cgo 调用栈 | C 函数调用 |

## 5. 垃圾回收（GC）

### 5.1 算法演进

| 版本 | 算法 | 特点 |
|------|------|------|
| Go 1.0~1.4 | 标记-清除（STW） | 简单，但停顿时间长 |
| Go 1.5 | **三色标记 + 并发** | 大幅降低 STW |
| Go 1.8+ | **混合写屏障** | 消除 STW 重扫描 |
| Go 1.12+ | 优化触发时机 | 更智能的 GOGC |

### 5.2 三色标记法

```
白色：未扫描（潜在垃圾）
灰色：已扫描自身，子对象待扫描
黑色：已扫描完成（存活对象）

扫描过程：
1. 所有对象初始为白色
2. 从根对象开始，标记为灰色
3. 取出灰色对象，扫描其引用，将引用对象标记为灰色，自身变黑
4. 重复直到无灰色对象
5. 剩余白色对象即为垃圾
```

### 5.3 并发 GC 阶段

```
┌─────────────────────────────────────────────────────────────┐
│                        GC 周期                               │
├─────────┬───────────────────────────────────┬───────────────┤
│ Mark    │         Concurrent Mark           │ Mark          │
│ Setup   │        （与应用并发执行）           │ Termination   │
│ (STW)   │                                   │ (STW)         │
├─────────┴───────────────────────────────────┴───────────────┤
│                      Sweep（并发清扫）                       │
└─────────────────────────────────────────────────────────────┘
     │                                              │
     └── STW 通常 < 1ms ──────────────────────────────┘
```

### 5.4 写屏障

**混合写屏障**（Go 1.8+）结合了插入屏障和删除屏障：

```go
// 伪代码：写屏障逻辑
func writePointer(slot *unsafe.Pointer, ptr unsafe.Pointer) {
    shade(*slot)  // 删除屏障：旧值变灰
    shade(ptr)    // 插入屏障：新值变灰
    *slot = ptr
}
```

### 5.5 GC 触发条件

| 触发方式 | 条件 |
|----------|------|
| 阈值触发 | 堆内存增长达到 GOGC 比例（默认 100%） |
| 定时触发 | 超过 2 分钟未 GC |
| 手动触发 | `runtime.GC()` |
| 内存压力 | 系统内存不足时 |

### 5.6 调优参数

```bash
# GOGC：触发 GC 的堆增长比例（默认 100）
GOGC=200 ./app  # 降低 GC 频率，增加吞吐

# GOMEMLIMIT（Go 1.19+）：软内存上限
GOMEMLIMIT=1GiB ./app  # 接近上限时更积极 GC

# 禁用 GC（特殊场景）
GOGC=off ./app
```

### 5.7 与 JVM GC 对比

| 维度 | JVM | Go |
|------|-----|-----|
| 分代 | 年轻代/老年代 | **无分代**（Go 1.x） |
| 算法选择 | 多种可选（G1/ZGC/Shenandoah） | 仅三色标记 |
| STW 时间 | ZGC < 10ms | 通常 < 1ms |
| 调优复杂度 | 高（数十个参数） | 低（主要 GOGC） |

## 6. 逃逸分析

### 6.1 定义

编译期静态分析，判断变量应分配在**栈**还是**堆**。

### 6.2 逃逸场景

| 场景 | 示例 | 分配位置 |
|------|------|----------|
| 返回局部变量指针 | `return &x` | 堆 |
| 闭包引用外部变量 | `func() { use(x) }` | 堆 |
| 接口动态类型 | `var i interface{} = x` | 可能逃逸 |
| 大对象 | `make([]byte, 1<<20)` | 堆 |
| 无法确定大小 | `make([]int, n)` | 堆 |

### 6.3 分析命令

```bash
# 查看逃逸分析结果
go build -gcflags="-m" main.go

# 更详细输出
go build -gcflags="-m -m" main.go
```

输出示例：
```
./main.go:10:6: moved to heap: x    # x 逃逸到堆
./main.go:15:2: y does not escape   # y 留在栈
```

### 6.4 优化建议

```go
// 避免逃逸的写法
func process(data []byte) {
    // 栈上分配
    var buf [1024]byte
    copy(buf[:], data)
}

// 会导致逃逸的写法
func process(data []byte) *[1024]byte {
    var buf [1024]byte
    copy(buf[:], data)
    return &buf  // buf 逃逸到堆
}
```

## 7. 同步原语

### 7.1 Channel

**CSP 模型核心**：通过通信共享内存，而非通过共享内存通信。

```go
ch := make(chan int, 10)  // 带缓冲
ch := make(chan int)      // 无缓冲（同步）

ch <- value  // 发送
v := <-ch    // 接收
close(ch)    // 关闭
```

### 7.2 sync 包

| 原语 | 用途 | 对标 Java |
|------|------|-----------|
| `sync.Mutex` | 互斥锁 | `synchronized` |
| `sync.RWMutex` | 读写锁 | `ReentrantReadWriteLock` |
| `sync.WaitGroup` | 等待一组 goroutine | `CountDownLatch` |
| `sync.Once` | 单次执行 | 双重检查锁定单例 |
| `sync.Cond` | 条件变量 | `Object.wait/notify` |
| `sync.Map` | 并发安全 Map | `ConcurrentHashMap` |

### 7.3 原子操作

```go
import "sync/atomic"

var counter int64
atomic.AddInt64(&counter, 1)
atomic.LoadInt64(&counter)
atomic.StoreInt64(&counter, 100)
atomic.CompareAndSwapInt64(&counter, old, new)
```

```go
    defer bufferPool.Put(buf)
    // 使用 buf
}
```

## 9. 总结：Go Runtime vs JVM

| 维度 | JVM | Go Runtime |
|------|-----|------------|
| 定位 | 跨平台虚拟机 | 嵌入式运行时库 |
| 执行模型 | 字节码解释 + JIT | AOT 编译 |
| 并发模型 | 1:1 线程 | M:N 协程（GMP） |
| 内存管理 | 分代 GC | 三色标记并发 GC |
| GC 停顿 | 可达 ms~s 级（取决于收集器） | 通常 < 1ms |
| 调优复杂度 | 高 | 低 |
| 启动速度 | 慢 | 快 |
| 生态定位 | 企业级、大型系统 | 云原生、微服务、CLI |
